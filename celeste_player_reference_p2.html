Climb State — Full Detail

StClimb
🧗
ClimbBegin — grabbing a wall ClimbGrabYMult = 0.2f ▶

When entering the climb state, vertical speed is heavily dampened, the player is nudged toward the wall, and a surface sound plays based on the wall material.

ClimbBegin()
private void ClimbBegin() {
    AutoJump        = false;
    Speed.X         = 0;
    Speed.Y        *= ClimbGrabYMult;   // 0.2 — kills most vertical momentum
    wallSlideTimer  = WallSlideTime;
    climbNoMoveTimer = ClimbNoMoveTime; // 0.1s — brief input lockout on grab
    wallBoostTimer  = 0;
    lastClimbMove   = 0;

    // Nudge player into wall (up to ClimbCheckDist = 2px)
    for (int i = 0; i < ClimbCheckDist; i++)
        if (!CollideCheck<Solid>(Position + Vector2.UnitX * (int)Facing))
            Position += Vector2.UnitX * (int)Facing;
        else break;

    // Surface-dependent grab sound
    var platform = SurfaceIndex.GetPlatformByPriority(
        CollideAll<Solid>(Position + Vector2.UnitX * (int)Facing, temp));
    if (platform != null)
        Play(Sfxs.char_mad_grab, SurfaceIndex.Param, platform.GetWallSoundIndex(this, (int)Facing));
}
ClimbGrabYMult = 0.2 is critical If you grab a wall while falling fast (say, 160px/s), you immediately get 0.2 × 160 = 32px/s of downward momentum. Without this, grabbing would feel like hitting a rubber wall — immediate full stop.
👆
SlipCheck — ledge detection SlipCheck(float addY) ▶

SlipCheck detects whether the player's hands are above the top of the wall. If so, they slip — which triggers a ClimbHop instead of continued climbing. This is how climbing over ledges works.

SlipCheck() — checks 4px above top of player hands
private bool SlipCheck(float addY = 0) {
    Vector2 at;
    if (Facing == Facings.Right)
        at = TopRight + Vector2.UnitY * (4 + addY);
    else
        at = TopLeft - Vector2.UnitX + Vector2.UnitY * (4 + addY);

    // True if BOTH checks are open — hands are past the ledge top
    return !Scene.CollideCheck<Solid>(at)
        && !Scene.CollideCheck<Solid>(at + Vector2.UnitY * (-4 + addY));
}

The addY parameter is used with -1 to check one pixel higher when going up, to detect the ledge slightly earlier and trigger a hop at the right moment.

When SlipCheck returns true during upward climbing, instead of stopping the player hits ClimbHop() — launching them over the ledge. This is what makes Celeste's climbing feel like natural scrambling rather than a rigid animation.

🐸
ClimbHop — scrambling over ledges ClimbHopY = -120f ▶

ClimbHop fires when the player reaches the top of a wall while climbing. It launches them upward and sideways, and if the wall they were holding is moving, it tracks that solid to avoid snapping away.

ClimbHop()
private void ClimbHop() {
    // Track the solid for smooth movement (handles moving platforms)
    climbHopSolid = CollideFirst<Solid>(Position + Vector2.UnitX * (int)Facing);
    playFootstepOnLand = 0.5f;

    if (climbHopSolid != null) {
        climbHopSolidPosition = climbHopSolid.Position;
        hopWaitX      = (int)Facing;
        hopWaitXSpeed = (int)Facing * ClimbHopX;  // 100 px/s
    } else {
        hopWaitX  = 0;
        Speed.X   = (int)Facing * ClimbHopX;      // immediate if no solid
    }

    Speed.Y         = Math.Min(Speed.Y, ClimbHopY); // -120 px/s cap, not force
    forceMoveX      = 0;
    forceMoveXTimer = ClimbHopForceTime;            // 0.2s
    noWindTimer     = ClimbHopNoWindTime;            // 0.3s — wind ignored briefly
}
hopWaitX — the moving platform trick When hopping off a moving solid, hopWaitX defers the horizontal speed assignment. Each frame the main update checks if the solid has moved, then tracks it, preventing the player from rubber-banding away from a platform mid-hop.
🎽
Wall Booster (Conveyor) — climb acceleration WallBoosterSpeed = -160f ▶

Some walls have a WallBooster entity attached. While climbing these, the player accelerates upward automatically — like a conveyor belt on a wall. Stamina is still drained.

WallBoosterCheck() + speed application
var booster = WallBoosterCheck();
if (climbNoMoveTimer <= 0 && booster != null) {
    wallBoosting = true;
    Speed.Y = Calc.Approach(Speed.Y, WallBoosterSpeed,    // -160 px/s
                             WallBoosterAccel * delta);    // 600 px/s²
    LiftSpeed = Vector2.UnitY * Math.Max(Speed.Y, WallBoosterLiftSpeed); // -80
    // Plays conveyor SFX loop with "end" parameter for smooth stop
}

The conveyor SFX uses an FMOD parameter called "end" which crossfades the looping audio to a stop sound when the player leaves the booster — rather than cutting abruptly.

Dream Dash — Full Detail

StDreamDash
💫
Dream Dash physics — naive movement TreatNaive = true ▶

Inside a dream block, the player moves with NaiveMove — bypassing all collision. Solids become passable. The trick is detecting when the player exits the dream block, then cleanly returning to normal collision.

DreamDashBegin()
private void DreamDashBegin() {
    Speed          = DashDir * DreamDashSpeed;   // 240 px/s
    TreatNaive     = true;                        // bypass solid collision
    Depth          = Depths.PlayerDreamDashing;    // render on top of dream blocks
    dreamDashCanEndTimer = DreamDashMinTime;    // 0.1s minimum time inside
    Stamina        = ClimbMaxStamina;            // refill on enter
    dreamJump      = false;
}
DreamDashUpdate() — each frame
private int DreamDashUpdate() {
    var oldPos = Position;
    NaiveMove(Speed * Engine.DeltaTime);   // ignores all solids

    if (dreamDashCanEndTimer > 0)
        dreamDashCanEndTimer -= delta;

    var block = CollideFirst<DreamBlock>();

    if (block == null) {          // exited the dream block
        if (DreamDashedIntoSolid()) {
            if (invincible) { Position = oldPos; Speed *= -1; }
            else Die(Vector2.Zero);
        } else if (dreamDashCanEndTimer <= 0) {
            // Freeze frame on exit
            Celeste.Freeze(.05f);
            // Jump if pressed + horizontal dash
            if (Input.Jump.Pressed && DashDir.X != 0) {
                dreamJump = true;
                Jump();
            }
            return StNormal;
        }
    } else {
        dreamBlock = block;
        // Displacement ripple inside block every 0.04s
        if (level.OnInterval(.04f)) {
            var d = level.Displacement.AddBurst(Center, .3f, 0f, 40f);
            d.WorldClipCollider = dreamBlock.Collider;
        }
    }
    return StDreamDash;
}
🧱
DreamDashedIntoSolid — wiggle-out logic DreamDashEndWiggle = 5 ▶

On exiting a dream block, the player might be partially inside a normal solid. The wiggle-out code tries all combinations of ±1..5 pixel offsets in X and Y, teleporting to the first clear position. If none work, the player dies.

DreamDashedIntoSolid()
private bool DreamDashedIntoSolid() {
    if (CollideCheck<Solid>()) {
        for (int x = 1; x <= DreamDashEndWiggle; x++)      // 1..5
          for (int xm = -1; xm <= 1; xm += 2)
            for (int y = 1; y <= DreamDashEndWiggle; y++)
              for (int ym = -1; ym <= 1; ym += 2) {
                Vector2 add = new Vector2(x * xm, y * ym);
                if (!CollideCheck<Solid>(Position + add)) {
                    Position += add;
                    return false;     // found a safe spot
                }
              }
        return true;   // truly stuck — die
    }
    return false;
}
This is what prevents soft-locks Without wiggle-out, clipping through thin geometry (a 2px wall for example) would leave the player stuck inside it. The 5-pixel wiggle covers nearly all cases. Invincibility mode bounces back instead of killing.
↗
DreamDashCheck — corner correction on entry DashCornerCorrection ▶

Before entering a dream block, if the player is partially blocked by a regular solid on the side, they can be nudged up to 4px perpendicular to the dash direction to slip into the dream block cleanly.

DreamDashCheck() — called from OnCollideH and OnCollideV
private bool DreamDashCheck(Vector2 dir) {
    if (Inventory.DreamDash && DashAttacking
        && (dir.X == Math.Sign(DashDir.X) || dir.Y == Math.Sign(DashDir.Y))) {

        var block = CollideFirst<DreamBlock>(Position + dir);
        if (block != null) {
            if (CollideCheck<Solid, DreamBlock>(Position + dir)) {
                // Blocked — try perpendicular nudge
                Vector2 side = new Vector2(Math.Abs(dir.Y), Math.Abs(dir.X));
                for (int i = -1; i >= -DashCornerCorrection; i--) {
                    var at = Position + dir + side * i;
                    if (!CollideCheck<Solid, DreamBlock>(at)) {
                        Position += side * i;
                        dreamBlock = block;
                        return true;
                    }
                }
                // Also try positive direction...
            } else { dreamBlock = block; return true; }
        }
    }
    return false;
}
03

Star Fly — Feather Flight

StStarFly
🪶
Speed lerp model — direction and magnitude separately StarFlyMaxLerpTime = 1f ▶

Star Fly separates speed into direction (rotated toward input) and magnitude (lerped toward a target). The target speed depends on whether you're pressing a direction, pressing the same direction as movement, or not pressing anything.

ConstantValueRole
StarFlyStartSpeed250 px/sInitial burst speed on feather entry
StarFlyTargetSpeed140 px/sSustained speed when pressing a direction
StarFlyMaxSpeed190 px/sMax speed when fully aligned with direction (after 1s)
StarFlySlowSpeed91 px/sSpeed when no input (drifts slowly)
StarFlyRotateSpeed320°/sHow fast direction rotates toward input
StarFlyMaxLerpTime1.0sTime to ramp from TargetSpeed to MaxSpeed
StarFlyAccel1000 px/s²Magnitude acceleration toward target speed
StarFlyTime2.0sTotal feather duration before expiring
StarFlyUpdate() — direction + speed logic
// Rotate current direction toward aim
Vector2 currentDir = Speed.SafeNormalize(Vector2.Zero);
if (currentDir == Vector2.Zero) currentDir = aim;
else currentDir = currentDir.RotateTowards(aim.Angle(),
                  StarFlyRotateSpeed * Engine.DeltaTime);

// Determine target speed based on alignment
float maxSpeed;
if (slow) {
    starFlySpeedLerp = 0;
    maxSpeed = StarFlySlowSpeed;              // no input
} else if (Vector2.Dot(currentDir, aim) >= .45f) {
    starFlySpeedLerp = Calc.Approach(starFlySpeedLerp, 1,
                       Engine.DeltaTime / StarFlyMaxLerpTime);
    maxSpeed = MathHelper.Lerp(StarFlyTargetSpeed,
                               StarFlyMaxSpeed, starFlySpeedLerp);
} else {
    starFlySpeedLerp = 0;
    maxSpeed = StarFlyTargetSpeed;             // pressing but not aligned
}

// Apply: approach magnitude, then multiply by direction
float currentSpeed = Speed.Length();
currentSpeed = Calc.Approach(currentSpeed, maxSpeed, StarFlyAccel * delta);
Speed = currentDir * currentSpeed;
Why separate direction and magnitude If you just lerped the whole velocity vector, sharp turns would first slow to zero (the shortest path through velocity space), creating a dead-stop mid-maneuver. Rotating direction separately while maintaining speed gives the feathery, continuous feel that Celeste's Star Fly is known for.
⏰
Star Fly exit — hitbox swap and speed clamp StarFlyReturnToNormalHitbox ▶

When the feather expires, the smaller star fly hitbox must be swapped back to the normal hitbox. If that hitbox is inside a solid, the code attempts two escape routes before throwing an exception.

HitboxSizeOffset
normalHitbox8 × 11-4, -11
starFlyHitbox8 × 8-4, -10
StarFlyReturnToNormalHitbox()
Collider = normalHitbox;
hurtbox  = normalHurtbox;

if (CollideCheck<Solid>()) {
    Vector2 start = Position;
    // Try moving up to clear the bottom edge difference
    Y -= (normalHitbox.Bottom - starFlyHitbox.Bottom);
    if (CollideCheck<Solid>()) {
        Position = start;
        // Try ducking + moving up
        Ducking = true;
        Y -= (duckHitbox.Bottom - starFlyHitbox.Bottom);
        if (CollideCheck<Solid>())
            throw new Exception("Could not get out of solids when exiting Star Fly!");
    }
}

// Speed clamped on exit
if (Speed.Y > StarFlyMaxExitY) Speed.Y = StarFlyMaxExitY;  // 0 — no downward exit speed
if (Math.Abs(Speed.X) > StarFlyMaxExitX)                   // 140
    Speed.X = StarFlyMaxExitX * Math.Sign(Speed.X);

The final speed clamp is important: exiting the feather while traveling at full StarFlyMaxSpeed (190) would give too much free momentum. The exit caps ensure the power-up doesn't trivialise sections that come after.

Swim State

StSwim
🌊
Water physics — three distinct zones SwimCheck / UnderwaterCheck ▶

Celeste's water has three detectable zones based on the player's position, each with different physics: surface, shallow, and fully underwater. Four raycast-equivalent checks handle the transitions.

The four water checks
private bool SwimCheck()          // both feet (-0) AND chest (-8) are submerged
    => CollideCheck<Water>(Position + Vector2.UnitY * -8)
    && CollideCheck<Water>(Position);

private bool SwimUnderwaterCheck() // head (-9) is submerged → fully under
    => CollideCheck<Water>(Position + Vector2.UnitY * -9);

private bool SwimJumpCheck()       // space 14px above → can jump out of water
    => !CollideCheck<Water>(Position + Vector2.UnitY * -14);

private bool SwimRiseCheck()       // 18px above is clear → auto-rise to surface
    => !CollideCheck<Water>(Position + Vector2.UnitY * -18);
ConstantValueRole
SwimMax80 px/sMax swim speed in any direction (surface)
SwimUnderwaterMax60 px/sMax horizontal speed when fully submerged
SwimMaxRise-60 px/sMax speed the player can rise naturally at surface
SwimAccel600 px/s²Acceleration in water
SwimReduce400 px/s²Deceleration when over max speed
SwimYSpeedMult0.5Vertical speed dampened on swim entry (SwimBegin)
SwimDashSpeedMult0.75Dash speed reduced by this when entering water mid-dash
↑
Auto-rise and surface pop SwimRiseCheck ▶

When the player is near the surface and not pressing down, they automatically rise. At the exact surface level (SwimCheck passes but SwimUnderwaterCheck fails), staying above water prevents sinking back in without input.

SwimUpdate() — surface rise + stay-at-surface
// Auto-rise: near surface, no vertical input
if (move.Y == 0 && SwimRiseCheck())
    Speed.Y = Calc.Approach(Speed.Y, SwimMaxRise, SwimAccel * delta);

// Stay at water surface (in Update, not SwimUpdate)
if (Speed.Y < 0 && Speed.Y >= SwimMaxRise) {
    while (!SwimCheck()) {
        Speed.Y = 0;
        if (MoveVExact(1)) break;   // nudge down 1px at a time
    }
}

// Ledge pop: shallow water, pressing sideways into a wall
const int ledgePopDist = -3;
if (!underwater && moveX != 0
    && CollideCheck<Solid>(Position + Vector2.UnitX * moveX)
    && !CollideCheck<Solid>(Position + new Vector2(moveX, ledgePopDist)))
    ClimbHop();   // pop up onto the ledge
The ledge pop Swimming into a wall that has a ledge 3px above the surface triggers a ClimbHop — the player automatically hauls themselves out. Without this, swimming into a ledge would just block movement. This is why Celeste's water sections don't feel frustrating at the edges.

Wind System

WindMover
💨
WindMove — per-frame push with many guards WindWallDistance = 3 ▶

Wind applies a direct position push each frame (not a speed change), but only after checking many conditions. It's physically blocked by walls, reduced when ducking, ignored in most special states, and suppressed briefly after a climb hop.

WindMove() — called by WindMover component
private void WindMove(Vector2 move) {
    // Guard: skip if any of these are true
    if (JustRespawned || noWindTimer > 0 || !InControl
        || StateMachine.State == StBoost
        || StateMachine.State == StDash
        || StateMachine.State == StSummitLaunch) return;

    // Horizontal wind
    if (move.X != 0 && StateMachine.State != StClimb) {
        windTimeout     = 0.2f;                        // for hair animation
        windDirection.X = Math.Sign(move.X);

        // Only push if 3px on the opposite side is clear
        if (!CollideCheck<Solid>(Position + Vector2.UnitX
                * -Math.Sign(move.X) * WindWallDistance)) {
            if (Ducking && onGround) move.X *= DuckWindMult; // 0 — no push when ducking
            // Clamp to level bounds
            if (move.X < 0) move.X = Math.Max(move.X, level.Bounds.Left - ExactPosition.X - Collider.Left);
            else              move.X = Math.Min(move.X, level.Bounds.Right - ExactPosition.X - Collider.Right);
            MoveH(move.X);
        }
    }

    // Vertical wind
    if (move.Y != 0) {
        windTimeout     = 0.2f;
        windDirection.Y = Math.Sign(move.Y);
        if (Speed.Y < 0 || !OnGround()) {
            if (StateMachine.State == StClimb) {
                if (move.Y > 0 && climbNoMoveTimer <= 0) move.Y *= .4f;
                else return;
            }
            MoveV(move.Y);
        }
    }
}
Wind is a position push, not a speed change This is intentional. If wind modified Speed, it would accumulate with player momentum, making the effect unpredictable. Directly calling MoveH/MoveV makes wind a fixed, consistent environmental force that doesn't compound. It also means wind stops immediately when conditions aren't met.

The windTimeout timer (set to 0.2s whenever wind fires) drives the hair animation — hair blows sideways while this is active, giving visual feedback that wind is currently affecting the player.

🔼
JumpThru boost — assisted upward slip JumpThruAssistSpeed = -40f ▶

When the player is rising and inside a one-way platform (JumpThru), they are gently pushed downward at -40px/s to slide out the bottom. Without this, rising into a JumpThru from below would cause the player to clip through and land on it — which is often undesirable during a fast upward movement.

Applied each frame in Update()
// Fires when: airborne, moving up, not climbing up, inside JumpThru, not blocked
if (!onGround && Speed.Y <= 0
    && (StateMachine.State != StClimb || lastClimbMove == -1)
    && CollideCheck<JumpThru>()
    && !JumpThruBoostBlockedCheck())
    MoveV(JumpThruAssistSpeed * Engine.DeltaTime);  // nudge: -40 × dt ≈ -0.67px/frame

JumpThruBoostBlockedCheck() checks if any LedgeBlocker component (used by strawberry seeds and some story objects) is preventing the through-movement, so the assist can be selectively disabled per entity.

OnSquish — Getting Crushed

OnSquish override
🫠
Duck-to-escape and TrySquishWiggle TrySquishWiggle ▶

When a moving solid tries to push the player into another solid (crush), Celeste first tries to duck the player to escape. If that fails, it tries to wiggle them to a nearby free position. Only if all escapes fail does the player die.

OnSquish() — called by the engine when a solid crushes the player
override protected void OnSquish(CollisionData data) {
    bool ducked = false;

    // Step 1: try ducking
    if (!Ducking) {
        ducked = true;
        Ducking = true;
        data.Pusher.Collidable = true;

        if (!CollideCheck<Solid>()) {
            data.Pusher.Collidable = false;
            return;   // ducking worked, player escaped
        }

        // Try target position while ducked
        var was = Position;
        Position = data.TargetPosition;
        if (!CollideCheck<Solid>()) {
            data.Pusher.Collidable = false;
            return;
        }
        Position = was;
        data.Pusher.Collidable = false;
    }

    // Step 2: try TrySquishWiggle (scans positions near the player)
    if (!TrySquishWiggle(data))
        Die(Vector2.Zero);     // all escapes failed — death
    else if (ducked && CanUnDuck)
        Ducking = false;        // managed to escape — stand back up
}
TrySquishWiggle is inherited from Actor The base Monocle Actor class provides TrySquishWiggle which scans a small area around the player (typically ±3px in each axis) looking for a free position. This is the same mechanism used by all Actor-derived entities in Celeste, not just the player.

⚠️
Killbox camera snap Killbox ▶

If a killbox (instant-death zone) is near and the player overlaps it horizontally, the camera is forced to show the danger zone — capped 180px above the killbox top. This ensures the player always sees what will kill them.

Killbox snap in CameraTarget
var killboxes = Scene.Tracker.GetEntities<Killbox>();
foreach (var box in killboxes) {
    if (!box.Collidable) continue;
    if (Top < box.Bottom && Right > box.Left && Left < box.Right)
        at.Y = Math.Min(at.Y, box.Top - 180);  // show 180px above the killbox
}
Design principle: never hide danger Celeste's level design depends on the player being able to see threats. The killbox snap is a safety net ensuring that screen-edge lava, death zones, and similar hazards are always within the viewport before the player reaches them.
🎯
CameraAnchor lerp — designer-controlled camera CameraAnchorLerp ▶

Level designers can set a CameraAnchor point and a CameraAnchorLerp strength (0..1 per axis). The camera target lerps toward the anchor by that amount each frame, letting cutscenes and scripted sequences gently pull the view toward points of interest.

Anchor lerp in CameraTarget getter
if (CameraAnchorLerp.Length() > 0) {
    if (CameraAnchorIgnoreX && !CameraAnchorIgnoreY)
        target.Y = MathHelper.Lerp(target.Y, CameraAnchor.Y, CameraAnchorLerp.Y);
    else if (!CameraAnchorIgnoreX && CameraAnchorIgnoreY)
        target.X = MathHelper.Lerp(target.X, CameraAnchor.X, CameraAnchorLerp.X);
    else if (CameraAnchorLerp.X == CameraAnchorLerp.Y)
        target = Vector2.Lerp(target, CameraAnchor, CameraAnchorLerp.X);
    else {
        target.X = MathHelper.Lerp(target.X, CameraAnchor.X, CameraAnchorLerp.X);
        target.Y = MathHelper.Lerp(target.Y, CameraAnchor.Y, CameraAnchorLerp.Y);
    }
}

Setting CameraAnchorLerp = Vector2(0.8, 0.8) on a target means 80% of the camera's "attention" goes to the anchor, and 20% follows the player. At 1.0, it's a hard lock on the anchor regardless of player position.

Chaser State — Badeline Mirror

ChaserState
👥
ChaserState recording — full snapshot every frame ChaserStateMaxTime = 4f ▶

Every physics frame, a ChaserState snapshot is appended to a rolling buffer. Badeline reads from this buffer with a time offset to mirror the player's exact movement from N seconds ago — including sounds, animations, hair color, and depth.

ChaserState struct — what is recorded
public struct ChaserState {
    public Vector2   Position;   // exact pixel position
    public float     TimeStamp;  // Scene.TimeActive when recorded
    public string    Animation;  // current animation ID
    public Facings   Facing;
    public bool      OnGround;
    public Color     HairColor;  // exact hair color including flash
    public int       Depth;

    // Up to 5 sounds that fired this frame (fixed array, no allocation)
    private ChaserStateSound sound0, sound1, sound2, sound3, sound4;
    public int Sounds;          // how many of the above are valid
}
Buffer management — UpdateChaserStates()
private void UpdateChaserStates() {
    // Evict entries older than ChaserStateMaxTime (4 seconds)
    while (chaserStates.Count > 0
        && Scene.TimeActive - chaserStates[0].TimeStamp > ChaserStateMaxTime)
        chaserStates.RemoveAt(0);

    // Append this frame's snapshot
    chaserStates.Add(new ChaserState(this));
    activeSounds.Clear();   // reset for next frame's sound collection
}
Reading back — GetChasePosition()
public bool GetChasePosition(float sceneTime, float timeAgo, out ChaserState chaseState) {
    foreach (var state in chaserStates) {
        float time = sceneTime - state.TimeStamp;
        if (time <= timeAgo) {
            // Accept if: within 20ms tolerance, or previous entry was too far back
            if (tooLongAgoFound || timeAgo - time < .02f) {
                chaseState = state;
                return true;
            } else { return false; }   // not enough history yet
        }
        else tooLongAgoFound = true;
    }
    chaseState = new ChaserState();
    return false;
}
Fixed-size sound array — no GC pressure The struct uses 5 inline fields (sound0–sound4) rather than a List. This avoids heap allocation every frame. A struct with 5 fixed fields is stack-allocated when passed around, keeping chaser state tracking effectively GC-free.
🔊
Sound replay system — Madeline→Badeline mapping MadelineToBadelineSound ▶

Each time the player plays a sound, the chaser system checks a lookup table. If a Badeline equivalent exists, it's stored in the current frame's ChaserState. Badeline then replays those sounds from her delayed position, making her feel like a true physical echo.

Sound recording in the Play() override
public FMOD.Studio.EventInstance Play(string sound, ...) {
    AddChaserStateSound(sound, param, value);   // record for chaser replay
    return Audio.Play(sound, Center, param, value);
}

private void AddChaserStateSound(string sound, ...) {
    string eventName = null;
    Sfxs.MadelineToBadelineSound.TryGetValue(sound, out eventName);
    if (eventName != null)      // only store if Badeline has an equivalent
        activeSounds.Add(new ChaserStateSound { Event = eventName, ... });
}

Loop sounds (wall slide, dream dash travel) use ChaserStateSound.Actions.Loop and .Stop actions in addition to oneshots, so Badeline's continuous sounds start and stop at exactly the right moments in her replay.

Trigger System

Triggers
🔲
OnEnter / OnStay / OnLeave — persistent tracking triggersInside HashSet ▶

Every frame, every trigger in the scene is tested. A HashSet tracks which triggers the player is currently inside, enabling proper Enter/Stay/Leave semantics. The set is also cleaned up on death and room transitions.

Trigger processing in Update()
foreach (Trigger trigger in Scene.Tracker.GetEntities<Trigger>()) {
    if (CollideCheck(trigger)) {
        if (!trigger.Triggered) {
            trigger.Triggered = true;
            triggersInside.Add(trigger);
            trigger.OnEnter(this);            // first frame inside
        }
        trigger.OnStay(this);               // every frame inside
    } else if (trigger.Triggered) {
        triggersInside.Remove(trigger);
        trigger.Triggered = false;
        trigger.OnLeave(this);              // first frame outside
    }
}
CleanUpTriggers() — called on death and transitions
public void CleanUpTriggers() {
    if (triggersInside.Count > 0) {
        foreach (var trigger in triggersInside) {
            trigger.OnLeave(this);    // fire OnLeave for all active triggers
            trigger.Triggered = false;
        }
        triggersInside.Clear();
    }
}
Why a HashSet, not a List The inner loop needs to check triggersInside.Contains() and .Remove() efficiently. A HashSet makes both O(1) rather than O(n). In a room with many triggers, this matters — the tracker query is O(n) but the Contains check is O(1).

Room Transitions

TransitionTo
🚪
BeforeDownTransition, BeforeUpTransition, TransitionTo OnTransition ▶

Room transitions are a carefully orchestrated process. Before the new room loads, the player's speed and state are adjusted to make the entry feel natural. During the transition, the player slides toward a target position. On completion, speed is snapped to clean integers.

BeforeDownTransition()If going down: set state to Normal, clamp Speed.Y to ≥0 (clear upward momentum), disable AutoJump and varJump. Disable any platform in the exit path.
BeforeUpTransition()Zero Speed.X. Set Speed.Y to JumpSpeed (-105). Enable AutoJump with VarJumpTime so the player continues rising into the new room. Set dashCooldownTimer = 0.2s.
TransitionTo(target, direction)Called each frame during transition. Moves 60px/s toward target. Updates hair and carry offset. Returns true when position == target.
OnTransition()Resets grace timer, force move, chaser states. Refills dash and stamina. Transfers followers (strawberries) to new room leader.
TransitionTo() — the slide-in movement
public bool TransitionTo(Vector2 target, Vector2 direction) {
    MoveTowardsX(target.X, 60f * Engine.DeltaTime);
    MoveTowardsY(target.Y, 60f * Engine.DeltaTime);

    // Still update hair even though normal update is suspended
    UpdateHair(false);
    UpdateCarry();

    if (Position == target) {
        ZeroRemainderX();  ZeroRemainderY();
        Speed.X = (int)Math.Round(Speed.X);  // snap to integer pixels
        Speed.Y = (int)Math.Round(Speed.Y);
        return true;
    }
    return false;
}
Speed snapping on transition complete Math.Round(Speed) ensures the player enters the new room with clean integer speeds. Sub-pixel floating point accumulated during the slide-in is stripped, preventing tiny drift that would compound across many transitions.

Respawn intro (StIntroRespawn) — detail
private void IntroRespawnBegin() {
    Depth    = Depths.Top;   // render on top of everything
    introEase = 1f;          // 1 = fully "dead", 0 = fully alive

    // Clamp the "death effect origin" to 40px inside level bounds
    Vector2 from = Position;
    from.X = MathHelper.Clamp(from.X, level.Bounds.Left + 40, level.Bounds.Right - 40);
    from.Y = MathHelper.Clamp(from.Y, level.Bounds.Top + 40, level.Bounds.Bottom - 40);
    deadOffset = from - Position;

    // 0.6s tween: deadOffset → 0, introEase → 0
    respawnTween = Tween.Create(Tween.TweenMode.Oneshot, null, .6f, true);
    respawnTween.OnUpdate = (t) => {
        deadOffset = Vector2.Lerp(from - Position, Vector2.Zero, t.Eased);
        introEase  = 1 - t.Eased;
    };
    respawnTween.OnComplete = (t) => StateMachine.State = StNormal;
}
Walk-in intro (StIntroWalk) — detail
public IEnumerator IntroWalkCoroutine() {
    var start = Position;
    // Place player off-screen
    if (IntroWalkDirection == Facings.Right) X = level.Bounds.Left - 16;
    else                                    X = level.Bounds.Right + 16;

    yield return .3f;           // brief pause before walking
    Sprite.Play(PlayerSprite.RunSlow);

    // Walk toward spawn until within 2px or blocked by solid
    while (Math.Abs(X - start.X) > 2
        && !CollideCheck<Solid>(Position + new Vector2((int)Facing, 0)))
    {
        MoveTowardsX(start.X, 64 * Engine.DeltaTime);
        yield return null;
    }

    Position = start;
    Sprite.Play(PlayerSprite.Idle);
    yield return .2f;
    StateMachine.State = StNormal;
}
