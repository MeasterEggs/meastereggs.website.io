00

Introduction

Celeste's Player.cs is one of the most-studied pieces of game feel engineering in indie game development. What makes it remarkable isn't any single trick — it's the density of small, compounding systems all working together. Every single jump input goes through at least six distinct processing stages before it becomes motion on screen.

This document walks through each system in detail, with the original C# constants, the reasoning behind them, and Godot 4 translation notes wherever the implementation differs.

CORE IDEA 01

Forgive timing

Coyote time, jump buffering, and corner correction all serve one goal: never punish the player for being almost right. Precision is required spatially, but not temporally.

CORE IDEA 02

Physics that feel physical

Gravity, friction, and momentum are tuned to feel weighty but controllable. The numbers aren't realistic — they're theatrical.

CORE IDEA 03

Every action confirmed

Squash/stretch, particles, sound variants, and rumble all fire on key actions. The player's brain registers the input as successful before the physics even resolve.

01

Architecture

The player runs a StateMachine with 23 states. Each state controls its own physics, animation, and collision rules. This isn't just organisation — it means physically different movement systems can coexist without giant if-chains.

State list (player.cs)
public const int StNormal        = 0;   // running, jumping, wall sliding
public const int StClimb         = 1;   // grabbing and climbing a wall
public const int StDash          = 2;   // dashing (normal)
public const int StSwim          = 3;   // water physics
public const int StBoost         = 4;   // inside a bubble booster
public const int StRedDash       = 5;   // red bubble dash
public const int StDreamDash     = 9;   // passing through dream blocks
public const int StStarFly       = 19;  // winged feather flight
// ... 15 more states for cutscenes, intros, special cases

The update loop each frame is:

1. Tick all timerscoyote, jump buffer, var jump, force move X, wall slide, wall boost…
2. Read ground stateCollideFirst<Solid> + CollideFirstOutside<JumpThru> one pixel below
3. Process inputmove_x, facing, aim direction, dash/jump press consumption
4. Run active state updateNormalUpdate, ClimbUpdate, DashUpdate, etc. Returns next state.
5. Apply movementMoveH / MoveV with collision callbacks
6. Update visualsSprite scale, hair, particles, camera, triggers
◈ Godot 4 note — state machine

Godot has a built-in AnimationStateMachine for animations, but for movement logic you'll want a manual integer variable (var state: int) and a match block, or an enum. The player.gd we wrote uses direct function dispatch for simplicity, which works identically. For larger games, consider Godot's Node-based state machine pattern (one child Node per state).

02

Coyote Time

Input Leniency
🐦
Jump Grace Timer jumpGraceTimer ▶

When you walk off a ledge without jumping, Celeste gives you a short window where a jump press still counts as a ground jump.

Named after the Looney Tunes character who runs off cliffs and only falls when he looks down. Without this, the player must jump before leaving the ledge, which feels arbitrarily punishing on fast movement.

Original C# constant
private const float JumpGraceTime = 0.1f;  // 100 milliseconds
How it's set and consumed
// Set each frame the player IS on the ground:
if (onGround)
    jumpGraceTimer = JumpGraceTime;
else if (jumpGraceTimer > 0)
    jumpGraceTimer -= Engine.DeltaTime;

// Consumed when jump pressed and grace window is open:
if (jumpGraceTimer > 0)
    Jump();
Key design insight The timer is set every frame on ground, not just when leaving. This means a long ledge-walk always has a fresh 100ms window at the very end. It also means the timer is effectively free when on the ground — it only becomes relevant the moment you leave.

Without coyote time

  • Must press jump before leaving ledge
  • Fast movement makes ledges feel random
  • Players blame themselves ("my fault")
  • Feels "slippery"

With coyote time

  • Jump near edge always works
  • Player feels in control at high speed
  • Only feels "wrong" if jump pressed very late
  • Feels responsive and forgiving
◈ Godot 4 implementation

In our player.gd: when was_on_ground && !on_ground && speed.y >= 0, set jump_grace_timer = JUMP_GRACE_TIME. The speed.y >= 0 guard prevents starting coyote time when jumping off a ledge intentionally (you'll be moving upward).

⏱
Jump Buffer (Pre-Input) Input.Jump.ConsumeBuffer() ▶

If you press jump slightly before landing, the input is stored and fires automatically the instant you touch the ground.

Celeste uses a general input buffering system (VirtualButton.ConsumeBuffer()). The buffer stores the last press time and checks if it's within a window. This is the companion to coyote time: coyote time is late jump, buffering is early jump.

Godot equivalent (player.gd)
## Set on button press:
if Input.is_action_just_pressed("jump"):
    jump_buffer_timer = JUMP_BUFFER_TIME   # 0.1s

## Consumed when conditions are met (any frame during landing):
if jump_buffer_timer > 0 and on_ground:
    jump_buffer_timer = 0
    _do_jump()
Why this matters Button press inputs in games are single-frame events. Without buffering, pressing jump 1-2 frames before landing is silently discarded. With buffering, the window expands to ~6 frames (0.1s at 60fps) — the player's intent is honoured rather than dropped.

Variable Jump Height

Input Leniency
📐
varJumpTimer + varJumpSpeed VarJumpTime = 0.2f ▶

Tapping jump gives a small hop. Holding jump gives a tall arc. The height is entirely determined by how long the button is held.

On the frame jump is pressed, varJumpSpeed stores the initial upward velocity. For the next 200ms, while the jump button is held, Speed.Y is clamped to that stored value. This means gravity is effectively paused during the rise as long as you hold the button.

Original constants
private const float JumpSpeed    = -105f;  // initial upward velocity
private const float VarJumpTime  = 0.2f;   // 200ms window
The variable jump logic in NormalUpdate
// Each frame while timer is running:
if (varJumpTimer > 0) {
    if (AutoJump || Input.Jump.Check)
        Speed.Y = Math.Min(Speed.Y, varJumpSpeed);  // prevent falling faster than jump-start speed
    else
        varJumpTimer = 0;                           // button released → stop extending
}
Speed.Y = min(current_speed_y, varJumpSpeed)
Clamping to varJumpSpeed prevents gravity from accelerating the player downward during the extension window. Releasing the button instantly allows gravity to take over.

Different jump types set varJumpTime differently to produce different arcs:

Jump typevarJumpTimevarJumpSpeedResult
Normal jump0.2f-105fFull controllable arc
Wall jump0.2f-105fSame as normal
Super wall jump0.25f-160fTaller, reward for technique
Bounce (spring)0.2f-140fSlightly higher than normal
Super bounce0.2f-185fVery high, from special blocks

Corner Correction

Input Leniency
🔲
Upward & Dash Corner Correction UpwardCornerCorrection = 4 ▶

When rising and nearly clearing a corner, the player is nudged sideways to slip through, instead of being stopped.

This applies during upward movement (hitting the ceiling while jumping) and during dashes. The code tries offsets from 1 to 4 pixels in each direction and teleports the player to the first valid position found.

Upward corner correction (OnCollideV)
// Called when rising and hitting a ceiling:
if (Speed.X <= 0) {
    for (int i = 1; i <= UpwardCornerCorrection; i++) {
        if (!CollideCheck<Solid>(Position + new Vector2(-i, -1))) {
            Position += new Vector2(-i, -1);
            return;   // nudged left — no collision!
        }
    }
}
// Mirror check for right side...
Dash corner correction
private const int DashCornerCorrection = 4;
private const int UpwardCornerCorrection = 4;

Without correction

  • 1px overlap = full stop
  • Jumps into blocks feel arbitrary
  • Dashing near walls clips unexpectedly

With correction (4px)

  • Near-miss becomes a clean pass
  • Jumps into gaps feel intended
  • Dashes hug walls cleanly
◈ Godot 4 note

Godot's move_and_slide() does some corner handling automatically via floor_snap_length. For ceiling correction specifically, you'll need to implement it manually inside your on_wall collision handling. Try: on collision while speed.y < 0, loop 1–4 px horizontal offsets and test test_move() at each offset.

Gravity Model

Physics
⬇
Single gravity constant, multiple multipliers Gravity = 900f ▶

Celeste uses one gravity constant but applies different multipliers depending on the situation. This produces distinct jump shapes without separate physics modes.

ConstantValuePurpose
Gravity900 px/s²Base downward acceleration, applied every frame
HalfGravThreshold40 px/sVertical speed below which half-grav activates at apex
MaxFall160 px/sTerminal velocity during normal falling
FastMaxFall240 px/sTerminal velocity when holding down (fast fall)
FastMaxAccel300 px/s²How quickly maxFall transitions to FastMaxFall
Gravity application (NormalUpdate vertical section)
// Multiplier: 0.5 at apex while holding jump, 1.0 otherwise
float mult = (Math.Abs(Speed.Y) < HalfGravThreshold
              && (Input.Jump.Check || AutoJump)) ? .5f : 1f;

// Also reduced in space areas:
if (level.InSpace) mult *= SpacePhysicsMult;  // 0.6f

Speed.Y = Calc.Approach(Speed.Y, max, Gravity * mult * Engine.DeltaTime);
Calc.Approach is not += Calc.Approach(current, target, step) moves current toward target by at most step, and never overshoots. This is important: it means gravity accelerates the player toward MaxFall, not past it. The equivalent in GDScript is move_toward(speed_y, max_fall, gravity * delta).
06

Apex Gravity

Physics
🌙
Half gravity at jump apex HalfGravThreshold = 40f ▶

Near the top of a jump arc, when vertical speed is very low, gravity is halved. This creates a hang-time effect that feels both satisfying and makes timing easier.

The condition checks both the speed threshold and that the jump button is held. This is deliberate: releasing jump cancels the variable jump extension, but also removes the half-grav benefit. The two systems work together.

if |speed_y| < 40 AND jump_held → gravity × 0.5
Only active when near apex (~40px/s vertical speed = about 4px of movement per frame at 60fps)

The threshold of 40px/s means half-gravity is active for roughly 4-6 frames around the apex at normal jump height. The player barely notices it consciously, but the jump arc feels more natural and readable.

◈ Godot 4 implementation

In our player.gd: if absf(speed.y) < HALF_GRAV_THRESH and Input.is_action_pressed("jump"): grav_mult = 0.5. This fires every frame in _handle_vertical(). Make sure it's evaluated before applying gravity: speed.y = move_toward(speed.y, max_fall, GRAVITY * grav_mult * delta).

07

Horizontal Jump Boost

Physics
➡
JumpHBoost on takeoff JumpHBoost = 40f ▶

Pressing a horizontal direction while jumping adds a flat 40px/s to horizontal speed. This makes jumps feel decisive and purposeful rather than purely vertical.

Applied in Jump()
Speed.X += JumpHBoost * moveX;   // moveX = -1, 0, or 1
Speed.Y = JumpSpeed;             // -105f

Note it's added to existing horizontal speed, not replacing it. If you're running at 90px/s and jump, you'll briefly reach 130px/s horizontally (90 + 40), which is above the normal max run speed. This is intentional — it makes running jumps feel fast and fluid.

Wall jumps use this differently: horizontal speed is set to a fixed value rather than boosted from current speed, giving wall jumps a consistent feel regardless of approach velocity.

Wall jump uses a fixed value instead
private const float WallJumpHSpeed = MaxRun + JumpHBoost;  // 90 + 40 = 130
// In WallJump():
Speed.X = WallJumpHSpeed * dir;   // always 130, regardless of current speed
08

Lift Speed

Physics
🏗
Platform momentum inheritance LiftSpeed / LiftBoost ▶

When jumping off a moving platform, the platform's velocity is added to the jump. Standing on a rising platform and jumping launches you higher; a descending one slightly weakens your jump.

LiftSpeed is set by the Solid (platform) as it moves. LiftBoost is a getter that clamps it to safe bounds before adding it to the jump velocity.

LiftBoost property
private Vector2 LiftBoost {
    get {
        Vector2 val = LiftSpeed;
        // Cap horizontal component
        if (Math.Abs(val.X) > LiftXCap)        // 250f
            val.X = LiftXCap * Math.Sign(val.X);
        // Only inherit upward platform movement
        if (val.Y > 0) val.Y = 0;               // ignore downward
        else if (val.Y < LiftYCap) val.Y = LiftYCap;  // -130f cap
        return val;
    }
}
The cap matters Without the Y cap, a very fast rising platform could launch the player through ceilings. LiftYCap = -130 means at most 130px/s of upward platform momentum is inherited, keeping it fun but safe.
09

Fast Fall

Physics
💨
Holding down increases max fall speed FastMaxFall = 240f ▶

Pressing down while airborne doesn't immediately slam you down — instead, the terminal velocity is raised, and you accelerate toward it over time.

Celeste gradually transitions maxFall from MaxFall (160) to FastMaxFall (240) using Calc.Approach with FastMaxAccel (300). This gives fast fall a smooth onset rather than a snappy instant change.

Fast fall logic in NormalUpdate
if (Input.MoveY == 1 && Speed.Y >= mf)  // mf = current max fall
    maxFall = Calc.Approach(maxFall, FastMaxFall, FastMaxAccel * delta);
else
    maxFall = Calc.Approach(maxFall, mf, FastMaxAccel * delta);

// Sprite squash when in fast fall territory
float half = mf + (fmf - mf) * .5f;
if (Speed.Y >= half) {
    float spriteLerp = Math.Min(1f, (Speed.Y - half) / (fmf - half));
    Sprite.Scale.X = MathHelper.Lerp(1f, .5f, spriteLerp);
    Sprite.Scale.Y = MathHelper.Lerp(1f, 1.5f, spriteLerp);
}

Notice the sprite squash activates at the halfway point between normal and fast max fall — so the visual feedback starts before you've reached max speed, giving the player a sense of building momentum.

Run Model

Physics
🏃
Asymmetric acceleration / deceleration RunAccel / RunReduce ▶

Acceleration toward max speed and deceleration from it use different constants, and air movement uses a multiplier. This is what gives movement its "weight" without feeling sluggish.

ConstantValueRole
MaxRun90 px/sHorizontal speed cap (normal)
RunAccel1000 px/s²Acceleration toward MaxRun (on ground)
RunReduce400 px/s²Deceleration when over max speed
AirMult0.65Both RunAccel and RunReduce scaled by this in air
DuckFriction500 px/s²Friction applied when crouching on ground
The two-branch acceleration logic
float mult = onGround ? 1 : AirMult;
// Over max and still going same direction → use RunReduce
if (Math.Abs(Speed.X) > max && Math.Sign(Speed.X) == moveX)
    Speed.X = Calc.Approach(Speed.X, max * moveX, RunReduce * mult * delta);
// Otherwise → use RunAccel (also handles braking and direction change)
else
    Speed.X = Calc.Approach(Speed.X, max * moveX, RunAccel * mult * delta);
Why RunReduce < RunAccel After a dash or speed boost, the player exceeds MaxRun. RunReduce (400) is slower than RunAccel (1000), so bleeding off excess speed takes longer than gaining it. This is what gives post-dash speed a lingering, satisfying "afterburn."

Wall Slide

Wall Mechanics
🧗
Progressive wall slide speed cap WallSlideTime = 1.2f ▶

Touching a wall while falling progressively reduces your fall speed from the normal max (160) down to a near-stop (20). The longer you're in contact, the more friction applies — like actual friction.

Wall slide max fall calculation
if (wallSlideDir != 0) {
    // Lerp: 1.0 (just touched) → 0.0 (slid for full WallSlideTime)
    max = MathHelper.Lerp(MaxFall, WallSlideStartMax,
                          wallSlideTimer / WallSlideTime);
    // Emit particles once you're settled into the slide (65% through)
    if (wallSlideTimer / WallSlideTime > .65f)
        CreateWallSlideParticles(wallSlideDir);
}
ConstantValueEffect
WallSlideStartMax20 px/sMax fall speed after sliding for WallSlideTime
WallSlideTime1.2sTime to reach full friction

The timer resets on landing and on any jump, so starting a new slide always begins with low friction and builds up.

Conditions for wall slide: pressing toward the wall, not holding down, holding object is null, not ducking, wall is within ClimbBoundsCheck (not at level edge), Speed.Y >= 0 (only sliding down, not rising).

Wall Jump

Wall Mechanics
🦘
Wall jump with force-move lock WallJumpForceTime = 0.16f ▶

Wall jumping locks horizontal input toward the jump direction for 160ms. Without this, a player pressing into the wall when they jump would immediately cancel the wall jump momentum.

Wall jump force move
private void WallJump(int dir) {
    if (moveX != 0) {
        forceMoveX      = dir;
        forceMoveXTimer = WallJumpForceTime;  // 0.16s
    }
    Speed.X = WallJumpHSpeed * dir;  // 130 px/s away from wall
    Speed.Y = JumpSpeed;             // -105 px/s upward
}
Force move override in horizontal processing
if (forceMoveXTimer > 0) {
    forceMoveXTimer -= delta;
    moveX = forceMoveX;   // player's actual input is ignored
} else {
    moveX = Input.MoveX.Value;
    climbHopSolid = null;  // also resets climb hop tracking
}

Wall jump check distance is only 3 pixels (WallJumpCheckDist = 3). This means the player can wall jump from up to 3px away from a wall — useful when sliding or when a small gap exists.

◈ Godot 4 — raycast vs CollideCheck

Celeste uses CollideCheck<Solid>(Position + Vector2.UnitX * dir * WallJumpCheckDist) — a simple AABB overlap test at an offset position. In Godot, use two RayCast2D nodes pointing left and right, or use test_move() with an offset transform. Raycasts are simpler for this use case.

Wall Speed Retention

Wall Mechanics
💾
Speed stored on wall hit, restored if wall clears WallSpeedRetentionTime = 0.06f ▶

When you run into a wall, your horizontal speed is zeroed — but stored for 60ms. If the wall disappears within that window (moving platform), your speed is restored. This prevents moving platforms from feeling like rubber walls.

OnCollideH stores speed
// Inside OnCollideH callback:
if (wallSpeedRetentionTimer <= 0) {
    wallSpeedRetained    = Speed.X;
    wallSpeedRetentionTimer = WallSpeedRetentionTime;  // 0.06s
}
Speed.X = 0;
Each frame, check if coast is clear
if (wallSpeedRetentionTimer > 0) {
    // Gave up — moved in opposite direction
    if (Math.Sign(Speed.X) == -Math.Sign(wallSpeedRetained))
        wallSpeedRetentionTimer = 0;
    // Wall gone — restore speed
    else if (!CollideCheck<Solid>(Position + Vector2.UnitX * Math.Sign(wallSpeedRetained)))
    {
        Speed.X = wallSpeedRetained;
        wallSpeedRetentionTimer = 0;
    }
    else
        wallSpeedRetentionTimer -= delta;
}

Wall Boost (Climb Jump)

Wall Mechanics
⬆
Sideways input after climb jump redirects momentum wallBoostTimer ▶

After a climb jump (jumping while grabbing a wall), if you quickly press sideways, the jump converts into a wall jump. This gives climb jumping a generous input window and prevents the player from just falling straight up.

Set on climb jump
if (moveX == 0) {  // no direction pressed at jump moment
    wallBoostDir   = -(int)Facing;
    wallBoostTimer = ClimbJumpBoostTime;  // 0.2s
}
Converted to wall jump if sideways input arrives
if (wallBoostTimer > 0) {
    wallBoostTimer -= delta;
    if (moveX == wallBoostDir) {
        Speed.X = WallJumpHSpeed * moveX;
        Stamina += ClimbJumpCost;   // refund stamina used
        wallBoostTimer = 0;
    }
}

Dash System

Dash
💥
Dash: speed, cooldowns, end speed DashSpeed = 240f ▶

The dash is a coroutine: it zeroes speed, waits one frame, then fires in the aimed direction at full speed. After DashTime (0.15s), it transitions back to Normal at a reduced EndDashSpeed.

t=0
DashBegin()

Speed = 0, freeze frame (0.05s), start cooldown timers, displacement burst

+1f
DashCoroutine starts

yield null (one frame). Then set Speed = aim × DashSpeed. This frame delay ensures Speed=0 is rendered once for a visual snap.

+0.15s
Dash ends

Speed = DashDir × EndDashSpeed (160). If upward component, multiply Y by EndDashUpMult (0.75). State → Normal.

+0.2s
DashCooldown expires

Player can dash again. Separate DashRefillCooldown (0.1s) controls when the dash crystal refills.

ConstantValueRole
DashSpeed240 px/sSpeed during the dash
EndDashSpeed160 px/sSpeed when transitioning back to Normal
EndDashUpMult0.75Upward speed multiplied by this at dash end
DashTime0.15sDuration of active dash

Super Jump / Hyper Jump

Dash
🎯
Dash → jump = high horizontal speed SuperJumpH = 260f ▶

Pressing jump while in a horizontal dash fires a super jump. The vertical is the same as a normal jump, but horizontal speed jumps to 260px/s — nearly 3× normal max run. This is Celeste's signature advanced technique.

SuperJump() method
private void SuperJump() {
    Speed.X = SuperJumpH * (int)Facing;  // 260 px/s
    Speed.Y = JumpSpeed;               // -105 px/s

    if (Ducking) {
        Ducking = false;
        Speed.X *= DuckSuperJumpXMult;   // 1.25 → 325 px/s (hyper jump!)
        Speed.Y *= DuckSuperJumpYMult;   // 0.5 → -52.5 px/s (lower arc)
    }
}
Hyper jump = ducked super jump If you're ducking while doing a super jump, X is multiplied by 1.25 (325px/s) but Y is halved (barely leaves ground). This is a skill-intensive technique for covering horizontal distance at maximum speed while staying low — the so-called "hyper dash."

Trigger condition (DashUpdate)

  • DashDir.Y == 0 (horizontal dash only)
  • CanUnDuck (not crouched unless hyper)
  • Input.Jump.Pressed
  • jumpGraceTimer > 0 (near ground)

Speed comparison

  • Normal run: 90 px/s
  • Super jump: 260 px/s
  • Hyper jump: 325 px/s
  • Post-dash: 160 px/s

Squash & Stretch

Visual Juice
🧃
Scale manipulation on key events Sprite.Scale ▶

The sprite is squashed and stretched on every major physical event. The scale is set to an extreme value and then smoothly approaches (1, 1) over time. The return speed is 1.75 units per second.

EventScale set toEffect
Jump takeoff(0.6, 1.4)Tall and thin — upward energy
Wall jump(0.6, 1.4)Same as jump
Direction flip (ducking)(0.8, 1.2)Subtle horizontal squash
Duck down(1.4, 0.6)Wide and flat — compressed
Stand up from duck(0.8, 1.2)Spring-back stretch
Landing (light)(1.0→1.6, 1.0→0.4)Wide squash, lerped by fall speed

Scale return in UpdateSprite (called every frame)
Sprite.Scale.X = Calc.Approach(Sprite.Scale.X, 1f, 1.75f * delta);
Sprite.Scale.Y = Calc.Approach(Sprite.Scale.Y, 1f, 1.75f * delta);
Landing squash (proportional to fall speed)
float squish = Math.Min(Speed.Y / FastMaxFall, 1);
Sprite.Scale.X = MathHelper.Lerp(1, 1.6f, squish);
Sprite.Scale.Y = MathHelper.Lerp(1, .4f, squish);
Scale, not animation These are all direct Sprite.Scale assignments — not keyframed animations. This means squash/stretch composites with whatever animation is playing, and works for any art style without extra keyframes.

Hair Physics & Color

Visual Juice
💜
Hair as a status indicator PlayerHair ▶

Madeline's hair color is a real-time HUD. It tells you how many dashes you have without any on-screen numbers. The hair also simulates physics, reacting to movement speed and wind.

StateHair colorMeaning
1 dash available■ #AC3232Normal red — default state
2 dashes available■ #FF6DEFPink — two-dash powerup active
0 dashes■ #44B7FFBlue — used both dashes
Dash used / flash■ WhiteFlash on change, then lerps to new color
Star fly (feather)■ #F2EB6DYellow — in feather flight mode

When dash count changes, the hair flashes white for 0.12s then transitions to the new color. This gives instant visual feedback without text or numbers.

Hair color logic in UpdateHair
if (lastDashes != Dashes) {
    color = FlashHairColor;       // White flash on change
    hairFlashTimer = .12f;
} else if (hairFlashTimer > 0) {
    color = FlashHairColor;
    hairFlashTimer -= delta;
} else if (Dashes == 2)
    color = TwoDashesHairColor;    // #FF6DEF
else
    color = NormalHairColor;       // #AC3232

The hair physics use a chain of nodes. Each frame, the physics step moves each node toward the previous with a damped spring, then the simulation walks the chain. Wind and high-speed movement directly offset Hair.StepPerSegment.

Hair node count also changes with dash count
Sprite.HairCount = (Dashes > 1 ? 5 : startHairCount);  // 5 or 4 nodes

State Machine

Systems
🔀
23-state machine with Begin/Update/End/Coroutine StateMachine ▶

Each state has up to four callbacks: Begin (runs once on enter), Update (runs every frame, returns next state), End (runs once on exit), and Coroutine (runs as a co-routine alongside Update).

State registration
StateMachine.SetCallbacks(StDash, DashUpdate, DashCoroutine, DashBegin, DashEnd);
//                         state   update fn   coroutine fn  begin fn  end fn

The coroutine pattern is particularly powerful. For the dash, a coroutine handles the timing (yield for DashTime, then set state to Normal). The Update handles per-frame checks like grabbing holdables or wall jumping during the dash. Both run simultaneously.

Why this matters for your game Without a state machine, you end up with massive if-chains: "if dashing AND not climbing AND not dead AND not in cutscene then...". A state machine makes each mode a clean, isolated unit. Adding a new movement mode (grapple hook, swim, glide) is adding one new state — not threading new conditions through every existing system.
◈ Godot 4 approach

For a Celeste-scale game, use an enum State and a match state: block in _physics_process. For complex states with their own coroutines, use Godot's func _some_state_coroutine() -> void with a running while and await get_tree().physics_frame. Store it in a variable so you can interrupt it when the state changes.

Camera

Systems
📷
Exponential lerp + state-based lookahead CameraTarget ▶

The camera uses an exponential lerp (not a fixed lerp) for frame-rate-independent smoothing. It also adds state-based offsets so the player always sees ahead of where they're going.

camera.pos += (target - camera.pos) × (1 - 0.01^deltaTime)
At 60fps: each frame moves ~7.4% of remaining distance. Approaches target asymptotically — never overshoots, slows as it arrives. Frame-rate independent.
Camera update (Player.Update)
var from   = level.Camera.Position;
var target = CameraTarget;
level.Camera.Position = from + (target - from)
    * (1f - (float)Math.Pow(0.01f, Engine.DeltaTime));
State-based lookahead offsets in CameraTarget getter
if (StateMachine.State == StStarFly) {
    target.X += .2f * Speed.X;   // leads 20% of speed
    target.Y += .2f * Speed.Y;
} else if (StateMachine.State == StRedDash) {
    target.X += 48 * Math.Sign(Speed.X);  // fixed 48px ahead
    target.Y += 48 * Math.Sign(Speed.Y);
} else if (StateMachine.State == StSummitLaunch) {
    target.Y -= 64;                        // shows what's above
}
◈ Godot 4 implementation

In Godot, apply the exponential lerp manually in _process(delta) on your Camera2D node. Set position_smoothing_enabled = false (disable Godot's built-in smoothing, which uses linear lerp). Then: camera.global_position += (target - camera.global_position) * (1.0 - pow(0.01, delta)). Add lookahead by adjusting target before this line based on player velocity.

Audio Feedback

Systems
🔊
Surface-aware audio, state variants, rumble SurfaceIndex ▶

Sound is tied to both action type and surface type. Footsteps sound different on stone vs grass. There are separate sounds for each jump variant. Controller rumble is calibrated per-action.

Surface-dependent footsteps (Sprite.OnFrameChange)
// On the correct animation frame for a footstep:
var landed = SurfaceIndex.GetPlatformByPriority(
    CollideAll<Platform>(Position + Vector2.UnitY, temp));
if (landed != null)
    Play(Sfxs.char_mad_footstep, SurfaceIndex.Param, landed.GetStepSoundIndex(this));
Sound eventVariants
JumpNormal, wall right, wall left, super, super wall, dream block, assisted
FootstepPer-surface via SurfaceIndex parameter (FMOD)
Wall slideLooping SFX, surface-parameterized, stops when not sliding
GrabPer-surface index on wall grabbed
Dash4 variants: red-left, red-right, pink-left, pink-right (by hair color + direction)
RumbleLight/Medium/Strong × Short/Medium; tied to jump type, dash, landing, climbing

The wall slide SFX is handled specially — it's a looping source that starts and stops based on the animation state, not a one-shot. The surface parameter is updated every frame mid-loop so it morphs as the player slides across different materials.

Tuning Guide

Reference

All of Celeste's constants are in the #region Constants block. Here's what to change for common feel problems:

"Jump feels too committed / punishing"
↑ JumpGraceTime (coyote)
↑ VirtualButton buffer time
"Jump height feels binary (full or nothing)"
↑ VarJumpTime (0.2 → 0.3)
Adjust HalfGravThreshold
"Air feels floaty / weightless"
↑ Gravity (900 → 1200)
↓ HalfGravThreshold
"Movement feels sluggish"
↑ RunAccel (1000 → 1600)
↓ AirMult towards 1.0
"Wall jump sends back into wall"
↑ WallJumpForceTime (0.16 → 0.22)
"Wall slide too slow / fast"
Adjust WallSlideStartMax (20)
and WallSlideTime (1.2)
"Post-dash speed drops too fast"
↓ RunReduce (400 → 250)
or ↑ EndDashSpeed (160)
"Squash/stretch too extreme"
Move scale values closer to (1,1)
e.g. (0.6,1.4) → (0.75,1.25)
"Corner correction clips through walls"
↓ UpwardCornerCorrection (4 → 2)
or ↓ DashCornerCorrection
Golden rule of feel tuning Change one constant at a time and playtest immediately. Feel is highly nonlinear — a 10% change in gravity feels entirely different than 10% in RunAccel, even though both affect "speed." Use the original Celeste values as your reference baseline and adjust from there.
